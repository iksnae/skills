# remotion-with-image

The composite skill: chain `image-generate` and `remotion-render` so
generated stills land inside a Remotion composition. If your project
keeps an image-voice section in a DESIGN.md/BRAND.md, the stills
inherit it automatically, so the final video is brand-aligned in both
static and motion layers without a separate brand pass.

For the underlying primitives (each bundles its own scripts under
`scripts/` in its skill directory; plugin installs keep the relative
layout):
- [`image-generate`](../image-generate/SKILL.md) — produces the PNGs
  (`scripts/generate_image.py`).
- [`remotion-author`](../remotion-author/SKILL.md) — encodes the
  BEATS-blueprint authoring discipline.
- [`remotion-render`](../remotion-render/SKILL.md) — invokes the
  render wrapper (`scripts/render_remotion.py`) and bundles the spec
  linter (`scripts/lint_remotion_spec.py`).

## Purpose and boundaries

This skill commits to:

- A single coherent workflow: generate → place under `public/` →
  reference via `staticFile()` → render.
- One natural location convention: generated stills land in
  `<remotion-project>/public/`. Remotion serves anything in
  `public/` at runtime, so `staticFile("hero.png")` resolves to
  `<project>/public/hero.png`.
- Reproducibility: the `image-generate` receipt JSON lives next to
  the PNG, so any rendered video has a complete audit trail of its
  static layers.

It does NOT commit to:

- Per-frame AI image generation (i.e. generating each frame with
  an image model mid-render). Remotion expects deterministic visuals
  for replayability; generated stills are pre-rendered and reused.
- Image editing or compositing inside Remotion beyond standard
  React props (opacity, transform, filter via `interpolate` /
  `spring`). For pixel-level manipulation, generate the modified
  version up front.
- Coordinating generation across many stills with shared state
  (e.g. "same character in five poses"). Image models don't have
  reliable cross-call consistency; a shared style brief is what
  carries continuity.

## Inputs

Required:

- **A Remotion project root** with `package.json` + `src/` and,
  ideally, a central design-tokens module (e.g. `brand.ts`).
- **One or more image prompts** — text descriptions for each still.
  The `image-generate` skill prepends any configured style brief
  automatically; the prompt body just describes what's in the image.
- **A composition spec** (BEATS-blueprint style) naming where each
  generated image lands in the timeline.

Optional:

- **`--props-file`** on the render — useful when the composition
  takes the image filename as a runtime prop (so the same
  composition can serve different stills).

Environment:

- **`OPENAI_API_KEY`** — required by the image-generate step.
- **`npx` (Node 16+) on PATH** — required by the render step.

## Workflow

### Step 1: Plan the stills

Before generating any image, draft the composition spec
([`remotion-author`](../remotion-author/SKILL.md) format). Identify
each scene that needs a still; note the slug + dimensions.

Don't generate stills speculatively. The image-generate cost is
~$0.04 per high-quality square; a batch of 20 unused stills is
~$0.80 of wasted imagery.

### Step 2: Generate the stills

```bash
python3 <image-generate-skill-dir>/scripts/generate_image.py \
  --prompt "<what's in the image>" \
  --size 1024x1024 \
  --quality high \
  --out <project>/public/<slug>.png
```

Any DESIGN.md/BRAND.md image-voice section is auto-injected (the
wrapper prepends it as a prompt prefix). Inspect the resulting PNG;
if anything in the brand layer surprises you, edit the style-brief
section rather than fighting the prompt.

The receipt lives beside the PNG. Keep it; it's the audit trail of
"this still came from this prompt at this time."

### Step 3: Reference in the composition

```tsx
import { staticFile, Img, AbsoluteFill, useCurrentFrame, interpolate } from "remotion";
import { BG, FONT_MONO } from "./brand";

export const StoryCard: React.FC = () => {
  const frame = useCurrentFrame();
  const fade = interpolate(frame, [0, 12], [0, 1], { extrapolateRight: "clamp" });

  return (
    <AbsoluteFill style={{ backgroundColor: BG }}>
      <Img
        src={staticFile("hero.png")}
        style={{
          width: "100%",
          height: "100%",
          objectFit: "cover",
          opacity: fade,
        }}
      />
      {/* caption overlay, motion via interpolate */}
    </AbsoluteFill>
  );
};
```

`staticFile()` resolves to `<project>/public/<filename>` at render
time. No absolute paths in source; the bundler handles it.

### Step 4: Lint the spec

```bash
python3 <remotion-render-skill-dir>/scripts/lint_remotion_spec.py \
  --spec <project>/specs/<clip>-spec.md
python3 <remotion-render-skill-dir>/scripts/lint_remotion_spec.py \
  --check-tokens <project>/src
```

First call enforces BEATS structure; second call catches any hex
literal that leaked outside the tokens module (default `brand.ts`).

### Step 5: Render

```bash
python3 <remotion-render-skill-dir>/scripts/render_remotion.py \
  --project <project> \
  --composition <StoryCardCompositionId> \
  --out renders/<clip>.mp4 \
  --tier default
```

The render wrapper does not download or modify the stills — they're
already on disk under `public/`. The receipt JSON captures
`composition`, `tier`, `codec`, output size, but not the still
provenance. For that, keep the image-generate receipts alongside
the project.

### Step 6: Verify

- Render exits 0, receipt `ok: true`.
- Output file size is sane (a 90-frame 1080p clip with one static
  image and a fade is usually 100–500KB at preview tier).
- Spot-check `--frames=0-1` to see the first frame in isolation —
  the still should be visible and the brand layer consistent.

## Cost shape

- **Image generation:** ~$0.04 per 1024×1024 high-quality still.
  Plan to generate each still once and reuse.
- **Render:** free. CPU + RAM bound, not API-bound.
- **Audit overhead:** ~3KB receipt JSON per image. Negligible.

A typical 30-second explainer with 5 generated hero shots and 3
recycled brand frames costs ~$0.20 in stills + zero in render.

## External-dependency failure modes

This pipeline depends on two externals, each with its own failure
shape. The skill is designed so transient failures degrade
gracefully and reported failures are honest about what broke.

| Stage | Depends on | Transient failure shape | Recovery |
|---|---|---|---|
| Image generation | OpenAI Images API | 429 rate-limit, 5xx, read/connect timeout | `generate_image.py` retries automatically (1× on 429 with reset-aware sleep; up to 2× on 5xx/timeout with 10s→30s backoff). Persistent failure → exit 1 + receipt with `error` field. |
| Composition load | Local filesystem + Remotion bundler | TS/JSX error in user code | Render exits 1; `stderr_tail` in the receipt carries the bundler error. |
| Composition discovery | `npx remotion compositions` | Network for first-run npx + bundler error | `render_remotion.py` pre-validates comp-IDs; surfaces the first error before the render starts. Pass `--no-validate-comp-id` to skip on repeated runs against a known project. |
| Render execution | Bundled headless Chrome | OOM, missing Chromium on first run, `delayRender()` timeout | First run downloads ~150MB Chromium; subsequent runs are fast. OOM → drop `--concurrency` or `--frames` slices. |
| Static-file resolution | Files under `<project>/public/` | Missing PNG (not generated, or generated to wrong path) | Render succeeds but the composition shows a broken `<Img>`. Spot-check `--frames=0-1` before committing the output. |

**Designing AROUND failure, not against it:**

- **Pre-generate, never per-render.** The image-gen step is the one
  with cost + flakiness. Treat the PNGs as committed assets, not
  build outputs. A render that re-generates stills every time is
  expensive and non-deterministic.
- **Receipts are the audit trail.** Both `generate_image.py` and
  `render_remotion.py` emit receipts. When a video looks wrong,
  read both receipts before re-running. The image-gen receipt says
  whether `style_injected` was true; the render receipt says
  whether the bundler errored.
- **Lint before render.** A 30-second 1080p render is cheap (CPU),
  but a render that ships with a hex literal leaking outside the
  tokens module is a brand bug. Run `lint_remotion_spec.py
  --check-tokens` in the same step as render.
- **Pipe-to-`tail` masks exit codes.** A smoke test like
  `tool --foo … | tail -3` ALWAYS shows exit 0 because `tail`
  succeeded on whatever it received. Use `${PIPESTATUS[0]}` or
  drop the pipe.

## Failure modes to avoid

- **Regenerating the same still on every render.** Image-generate is
  not idempotent — the same prompt produces different pixels each
  call. Generate once, commit the PNG, reuse forever. If a still
  needs updating, treat it as a deliberate brand evolution.
- **Hardcoding `public/` paths.** Always use `staticFile()`. Direct
  string paths break under Remotion's bundler.
- **Per-frame stills.** Generating 1800 stills for a 60-second 30fps
  clip costs ~$72 and produces inconsistent visuals frame-to-frame.
  Compose motion in Remotion against a small set of well-chosen
  stills.
- **Bypassing the style brief.** Passing `--no-style` to
  image-generate produces an off-brand still even if the composition
  that uses it is on-brand. The still looks pasted-in. Render
  brand-aware unless the still is a chart/diagram where brand voice
  would distract.
- **Skipping the spec.** "Just generate three stills and put them in
  a composition" produces unreviewable output. Author the spec
  first ([`remotion-author`](../remotion-author/SKILL.md)); the
  spec names which images are needed before any are generated.

## Verification

The pipeline produced a usable artifact when:

- Every still referenced via `staticFile()` exists on disk under
  `<project>/public/`.
- Each still has a sibling receipt JSON with `ok: true` (and
  `style_injected: true` when a style brief is configured).
- The composition's `.tsx` does not hardcode hex colors
  (`--check-tokens` linter clean).
- The render receipt has `ok: true` and `exit_code: 0`.
- Visual inspection: the brand layer reads consistent between the
  static image and the motion overlay (same palette, same
  typography, same feel).
